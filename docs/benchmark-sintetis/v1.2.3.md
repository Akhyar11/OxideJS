## Perubahan Penting Dibanding Versi Sebelumnya

- Benchmark recurrent utama tidak lagi memproses sample satu per satu di dalam batch efektif.
- `Embedding` mendapat jalur `forwardTimeMajor(...)` untuk recurrent batching.
- `RNN`, `LSTM`, dan `GRU` mendapat hot path `forwardBatch(...)` / `backwardBatch(...)`.
- Input projection recurrent sekarang diprecompute lebih besar di jalur batch, sehingga loop scalar JavaScript di hot path turun signifikan.
- Snapshot ini juga mendokumentasikan correctness companion secara eksplisit, bukan hanya angka performa.

## Interpretasi

- Kenaikan throughput terbesar datang dari Tier 1: representasi batch recurrent yang valid dan benchmark yang sekarang fair.
- `RNN` menjadi model tercepat pada workload ini karena struktur sel paling sederhana dan overhead gate paling kecil.
- `GRU` dan `LSTM` juga naik tajam karena input projection dan backward batch sekarang lebih matrix-heavy.
- Benchmark acuan utama untuk perubahan ini adalah `math_synthetic_baseline`, tetapi interpretasinya harus dibaca bersama `correctness_companion`.
- Benchmark ini sudah jauh lebih fair daripada snapshot sebelumnya, tetapi jalur batch baru masih dipakai oleh harness benchmark recurrent, bukan oleh `Sequential.fit()` generic.

## Tindak Lanjut

- Tambahkan correctness suite khusus untuk `forwardBatch(...)` / `backwardBatch(...)` bila path ini ingin dianggap contract internal yang stabil.
- Pertimbangkan penggabungan gate weight menjadi satu matmul besar untuk `LSTM` dan `GRU` agar jumlah projection full-sequence turun lagi.
- Pertimbangkan pemindahan hot path recurrent batch ke Rust/native backend untuk mengurangi overhead JS yang tersisa.
- Pertahankan command benchmark dan subset data ini agar histori tetap comparable.

## Versioning

Versi aktif proyek saat ini adalah `1.2.3`.

Proyek ini memakai format versi `MAJOR.MINOR.PATCH` seperti `1.2.3`.

- Angka paling depan (`MAJOR`): perubahan besar yang biasanya membawa breaking change atau perubahan arsitektur utama.
- Angka tengah (`MINOR`): penambahan fitur baru atau peningkatan yang tetap kompatibel dengan versi sebelumnya.
- Angka paling belakang (`PATCH`): perbaikan bug, optimasi kecil, cleanup, atau perubahan minor yang tidak mengubah API utama.

Contoh:
- `1.2.3`: rilis mayor `1`, minor `2`, patch `3` untuk optimasi batch recurrent.
- `1.1.4`: masih di mayor `1` dan minor `1`, tetapi sudah ada 4 patch/perbaikan kecil dari baseline `1.1.0`.
